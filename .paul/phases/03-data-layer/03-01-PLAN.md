<objective>
## Goal
Create the composable data layer — 3 files per type (guide.md, config.md, skill-loadout.md) across all 5 default types, totaling 15 files.

## Purpose
This is what makes SEED composable. The ideate task loads these files after type selection to shape the conversation. Without them, ideate has no type-specific knowledge. The data layer is the reason SEED can handle 5 project types without 5 separate task files.

## Output
- 15 files across data/{type}/ directories (3 per type × 5 types)
</objective>

  <action>
    Create 15 files across 5 type directories. Each type gets 3 files:

    **guide.md format (per type):**
    ```markdown
    # {Type} — Conversation Guide

    ## Section 1: {Name}
    **Explore:** {What to ask / discuss}
    **Suggest:** {What to offer if user is stuck}
    **Depth:** {required | optional}

    ## Section 2: {Name}
    ...
    ```
    - Application: 10 sections (Problem, Stack, Data Model, API, Deployment, Security, UI/UX, Integrations, Phases, Skill Loadout)
    - Workflow: 8 sections (Problem, Scope, Integration Map, Interaction Design, Output Artifacts, Downstream Handoff, Tool Discovery, Edge Cases)
    - Client: 7 sections (Business Context, Competitors, Conversion, Content, Tech, Timeline, Skill Loadout)
    - Utility: 6 sections (Problem, Scope Guard, User & Distribution, Dependencies, Interface, Done Criteria)
    - Campaign: 7 sections (Strategy, Audience, Deliverables, Channels, Timeline, Metrics, Skill Loadout)

    Each section must have BOTH "Explore" (questions/prompts) AND "Suggest" (coaching guidance). The existing /ideate command has bare section headers — expand them into actual coaching content.

    **config.md format (per type):**
    ```markdown
    # {Type} — Configuration

    | Setting | Value |
    |---------|-------|
    | Rigor | {tight/standard/deep/creative} |
    | Demeanor | {description} |
    | Sections | {count} |

    ## Required Sections
    - {list}

    ## Optional Sections
    - {list}
    ```

    Rigor mapping:
    - application → deep (thorough, architecture matters)
    - workflow → standard (balanced exploration)
    - client → standard (balanced, business-focused)
    - utility → tight (fast, resist scope creep)
    - campaign → creative (loose, generative, timeline-focused)

    **skill-loadout.md format (per type):**
    ```markdown
    # {Type} — Skill Loadout

    ## Recommended Tools

    | Tool | Priority | Why |
    |------|----------|-----|
    | {tool} | required/recommended/optional | {reason} |
    ```

    Tool mapping:
    - application: PAUL (required), AEGIS (recommended), ui-ux-pro-max (recommended), sonarqube (optional)
    - workflow: PAUL (required), Skillsmith (recommended), CARL (recommended)
    - client: PAUL (recommended), business-context-extractor (required), revops-expert (recommended), ui-ux-pro-max (recommended)
    - utility: PAUL (optional — may be overkill), AEGIS (optional)
    - campaign: content-flywheel-strategist (recommended), revops-expert (recommended), content-manager (recommended)

    **What to avoid:**

    - Do NOT make all types identical — rigor, section count, and demeanor should vary meaningfully
    - Do NOT invent tools that don't exist in Chris's ecosystem — use real skill names from the workspace
