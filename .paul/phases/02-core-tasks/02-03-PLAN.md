<objective>
## Goal
Create `tasks/launch.md` — a thin wrapper that runs the graduate flow then detects and initializes PAUL, giving users a single command to go from ideation to managed build.

## Purpose
Launch is the "easy button." Instead of running `/seed graduate` then manually running `/paul:init`, the user runs `/seed launch` and gets both in one flow. The key design decision: PAUL init is headless — SEED's ideation already captured everything PAUL would ask, so PLANNING.md is passed directly without re-asking questions.

## Output
- `tasks/launch.md` — Skillsmith-compliant task file (~100-130 lines)
</objective>

  <action>
    Create tasks/launch.md following Skillsmith task rules (6 XML sections, no frontmatter, snake_case step names).

    **Structure the file with these steps:**

    1. `run_graduation` (priority="first") — Delegate to tasks/graduate.md by referencing it. State clearly: "Execute the full graduation flow from tasks/graduate.md first. All validation, quality checks, README synthesis, git init, and tracking updates happen there. Do NOT duplicate that logic here." After graduation completes, proceed to PAUL integration.

    2. `offer_paul` — Ask the user: "Want to initialize PAUL for a managed build? PAUL gives you structured phases, plans, and progress tracking." Wait for response. If user declines: report graduation complete, suggest they can run `/paul:init` later if they change their mind. Exit.

    3. `check_paul_availability` — Check if PAUL is available (paul-framework installed). The check: look for `~/.claude/paul-framework/` directory existence.
       - If available: proceed to init
       - If not available: "PAUL isn't installed globally yet. Want to install it?" Provide installation guidance. Wait for response. If user installs, proceed. If user declines, exit gracefully.

    4. `headless_paul_init` — Initialize PAUL with headless context:
       - Read the PLANNING.md from the graduated project
       - Pass it as context to `/paul:init` with instruction: "Use this PLANNING.md as the project brief. Derive milestones, phases, and structure from it. Do NOT re-ask questions that SEED already answered. Propose the project structure and ask for approval."
       - The user still reviews and approves the PAUL structure — headless means no questions, not no approval
       - Wait for response (PAUL init approval)

    5. `report_completion` — Report full launch result:
       ```
       Launched: {name}
       Location: apps/{name}/
       PAUL: Initialized with {N} phases

       You're ready to build. Run /paul:plan to start Phase 1.
       ```

    **What to avoid and WHY:**
    - Do NOT copy graduate.md logic into launch.md — launch delegates to graduate, it doesn't duplicate it. This is the wrapper pattern.
    - Do NOT auto-accept PAUL structure — headless means no questions, but the user still approves the proposed structure
    - Do NOT scan the filesystem for PAUL — ask the user, then check if they said yes
    - Keep this file thin (~100-130 lines) — launch is a wrapper, not a standalone flow
