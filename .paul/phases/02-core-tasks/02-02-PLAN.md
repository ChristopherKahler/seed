<objective>
## Goal
Create `tasks/graduate.md` — the task that moves a completed ideation from `projects/{name}/` into `apps/{name}/` with git init and README synthesis.

## Purpose
Graduate is SEED's handoff mechanism. It takes the PLANNING.md produced by ideate and turns it into a real project directory. This is the bridge between "I have a plan" and "I have a repo." Without it, ideation output sits idle in projects/ forever.

## Output
- `tasks/graduate.md` — Skillsmith-compliant task file (~120-150 lines)
</objective>

  <name>Task 1: Create tasks/graduate.md with full Skillsmith-compliant structure</name>
  <files>tasks/graduate.md</files>
  <action>
    Create tasks/graduate.md following Skillsmith task rules (6 XML sections, no frontmatter, snake_case step names).

    **Structure the file with these steps:**

    1. `validate_input` (priority="first") — Check $ARGUMENTS for project name. Validate: projects/{name}/PLANNING.md exists, apps/{name}/ does NOT exist. Abort with clear messages on failure. If no argument, list available projects in projects/ that have PLANNING.md files and ask user to pick.

    2. `quality_check` — Read PLANNING.md. Reference checklists/planning-quality.md to assess if content is rich enough for graduation. Extract Type metadata. If quality is insufficient, warn user: "This PLANNING.md may be too thin for a clean graduation. Want to go back to /seed to flesh it out, or proceed anyway?" Wait for response.

    3. `create_app_directory` — Create apps/{name}/ directory and initialize git repo (git init -b main).

    4. `synthesize_readme` — Read PLANNING.md content. Generate type-aware README.md:
       - Common sections for all types (title, description, type, skill loadout, overview, design decisions, phases, open questions)
       - Type-specific sections based on Type metadata (adapt from existing graduate command's README templates)
       - Synthesis guidelines: remove brainstorming artifacts, keep resolved decisions, keep architecture details, keep phases
       - The README should be comprehensive enough to serve as a PAUL-ready project brief
       - Present README to user for review. Wait for response.

    5. `finalize` — Create initial git commit. Update ACTIVE.md with graduated location. Append graduation note to original projects/{name}/PLANNING.md. Report completion with next step options (/paul:init or /seed launch).

    **What to avoid and WHY:**
    - Do NOT include the full type-specific README templates inline — the existing graduate command has ~80 lines of README templates. Keep them as inline guidance (what sections per type) but not full markdown templates. The task is instructions for Claude, not a template engine.
    - Do NOT make this task depend on ideate.md having run — graduate should work on ANY projects/{name}/PLANNING.md, even manually created ones
    - Do NOT include PAUL initialization — that's launch.md's job (Plan 02-03)
    - Follow the same wait-point and step-naming patterns established in tasks/ideate.md
  </action>
