## Goal
Create `tasks/ideate.md` — SEED's primary task file. Implements type-first guided ideation that takes a raw idea through collaborative exploration and produces a populated PLANNING.md document.

## AC-1: Skillsmith Task Compliance
```gherkin
Given the generated tasks/ideate.md file

Then all 6 required XML sections are present in order (purpose, user-story, when-to-use, steps, output, acceptance-criteria)
And no YAML frontmatter exists
And all steps have snake_case name attributes

```

## AC-2: Type-First Flow
```gherkin
Given a user invokes /seed or /seed ideate
When the task begins execution
Then type determination happens BEFORE any substantive ideation
And if $ARGUMENTS matches a known type, it routes directly
And if $ARGUMENTS is empty/unknown, a discovery conversation determines type
And type selection is confirmed before proceeding
```

## AC-3: Composable Data Loading
```gherkin
Given a type has been selected (e.g., "application")
When the ideation conversation begins
Then data/{type}/guide.md is referenced for conversation sections
And data/{type}/config.md is referenced for rigor/demeanor settings
And the task does NOT hardcode type-specific content — it delegates to data files
```

## AC-4: Wait Points at User Input
```gherkin
Given the task is executing
When user input is genuinely needed (type confirmation, name confirmation, section exploration, readiness signal)
Then explicit "Wait for response" signals are present
And the task does NOT race through sections without user interaction
```

## AC-5: Coach Persona Embedded
```gherkin
Given the task instructions
When Claude executes the ideation flow
Then the tone is collaborative (brainstorming alongside, offering suggestions)
And NOT interrogative (firing questions, demanding answers)
And rigor adapts: tight for utilities, deep for applications, creative for campaigns
```

## AC-6: PLANNING.md Output
```gherkin
Given ideation is complete (user signals readiness)
When the task generates output
Then it reads templates/planning-{type}.md for structure
And populates it with conversation content
And writes to projects/{name}/PLANNING.md
And the output is rich enough for headless PAUL init (per checklist reference)
```

## AC-7: Existing Project Detection
```gherkin
Given a user provides a project name
When projects/{name}/ already exists
Then the task asks whether to continue existing ideation or pick a different name
And does NOT silently overwrite
```

  <name>Task 1: Create tasks/ideate.md with full Skillsmith-compliant structure</name>
  <files>tasks/ideate.md</files>
  <action>
    Create tasks/ideate.md following Skillsmith task rules (6 XML sections, no frontmatter, snake_case step names).

    **Structure the file with these steps:**

    1. `determine_type` (priority="first") — Check $ARGUMENTS for type match (application|app, workflow, client, utility|tool, campaign|content). If match, confirm and proceed. If no match, ask user to describe their idea, suggest best-fit type with reasoning, confirm. Reference the type signals table from the existing ideate command.

    2. `get_project_name` — Ask for lowercase hyphenated name. Check `projects/{name}/` existence. If exists, offer to continue or rename. Create `projects/{name}/` directory.

    3. `load_type_context` — Read `data/{type}/guide.md` for conversation sections. Read `data/{type}/config.md` for rigor level and demeanor. This step sets the tone for the rest of the session. Include a brief coach-persona instruction: "You are a project coach. Brainstorm alongside the user, offer concrete suggestions when they're stuck, push toward decisions when it's time, let ideas breathe when they need space."

    4. `guided_conversation` — Work through sections from guide.md. Key instructions:
       - These are conversation prompts, NOT a questionnaire
       - Adapt based on discussion flow
       - Skip sections the user has already addressed naturally
       - Circle back to sections that need more depth
       - Offer concrete suggestions (e.g., "For this type of app, most people use X because...")
       - Use "Wait for response" after presenting each section group
       - Respect config.md rigor level: tight types get fewer sections, deeper types get more exploration

    5. `generate_planning` — When user signals readiness (or all sections covered):
       - Read `templates/planning-{type}.md`
       - Populate with conversation content
       - Reference `checklists/planning-quality.md` as quality gate
       - Write to `projects/{name}/PLANNING.md`
       - Wait for response — let user review

    6. `update_tracking` — Update ACTIVE.md with new project entry (location, type, status: "Ideation complete — ready for /seed graduate"). Report completion with next step suggestion.

    **What to avoid and WHY:**
    - Do NOT hardcode the 5 type guides inline — the existing /ideate command does this and it's 120+ lines of type-specific content. Instead, delegate to `data/{type}/guide.md` via @-reference. This is the composable data layer decision.
    - Do NOT use BDD Given/When/Then in acceptance-criteria section — Skillsmith uses plain `- [ ]` checklists
    - Do NOT skip wait points — the existing command has no wait points and races through
    - Do NOT copy the existing /ideate command verbatim — adapt its logic into Skillsmith structure with composable data loading
